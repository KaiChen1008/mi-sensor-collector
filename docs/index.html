2 Core Data Flow

每隔 SCAN_INTERVAL_SECONDS（預設 60 秒）掃描一次，每次分 3 個階段。

階段 ① — BLE 讀取與解析

GATT characteristic UUID ebe0ccc1-7a0a-4b0c-8a1a-6ff2997da3a6 回傳 5 bytes：

# backend/app/services/ble_scanner.py
data = await client.read_gatt_char(CHAR_UUID)
temp     = int.from_bytes(data[0:2], signed=True) / 100   # e.g. 2650 → 26.50°C
humidity = data[2]                                          # e.g. 65 → 65%
voltage  = int.from_bytes(data[3:5]) / 1000               # e.g. 2800 → 2.8V
battery  = max(0, min(100, int((voltage - 2.1) / 1.0 * 100)))

沒有 BLE 硬體時，SIMULATE_SENSORS=true 會改呼叫 _simulate_reading() 回傳隨機數值。

階段 ② — Alert Engine

# backend/app/services/alert_engine.py
for rule in rules:
    value = _metric_value(reading, rule.metric)   # temperature / humidity / battery
    compare = OPERATORS[rule.operator]            # >, <, >=, <=, ==, !=
    if not compare(value, rule.threshold):
        continue
    if elapsed < timedelta(minutes=rule.cooldown_minutes):
        continue                                  # 還在 cooldown 內，跳過
    await _fire_alert(db, rule, reading, sensor, value, now)

階段 ③ — WebSocket Broadcast

ConnectionManager 維護 list[WebSocket]，自動清掉拋出例外的斷線 client：

# backend/app/main.py
async def broadcast(self, payload: dict) -> None:
    message = json.dumps(payload)
    dead = []
    for ws in self._connections:
        try:
            await ws.send_text(message)
        except Exception:
            dead.append(ws)
    for ws in dead:
        self._connections.remove(ws)

重點設計

設計說明
alert_rules.sensor_id = NULL全域規則，套用到所有 sensor，不需要重複建立 N 條
alert_logs append-onlynotification_sent=false + error_message 可追蹤失敗原因
readings.timestamp 有 index讓歷史查詢按時間範圍篩選高效
無 migration 工具schema 改了就刪 backend/data/sensors.db，由 init_db() 重建

欄位細節

sensors

  • ble_address：macOS 上是 UUID 字串（不是 MAC address），bleak 在 macOS 的行為如此
  • is_active = false：BLEScanner 掃描時跳過，但資料與 rules 保留

alert_rules

  • metric：temperature | humidity | battery
  • operator：> < >= <= == !=
  • channel_target：email 填 email address；telegram 填 numeric chat ID；line 填 personal access token
  • cooldown_minutes：預設 30 分鐘，防止通知洗版

alert_logs

  • notification_sent = false + error_message 記錄送出失敗的原因
  • append-only，不刪不改，是完整的通知歷程

Global Singleton — 避免 Circular Import

main.py 無法直接把 manager.broadcast 傳給 ble_scanner（會 circular import），改用 module-level 變數：

# backend/app/services/ble_scanner.py
_scanner_instance: "BLEScanner | None" = None  # API routes 用這個 trigger 手動掃描
_ws_broadcast_fn = None                         # main.py 啟動後注入

# backend/app/main.py — lifespan()
scanner = ble_scanner.BLEScanner()
ble_scanner._scanner_instance = scanner                   # 暴露給 API routes
ble_scanner.register_broadcast(manager.broadcast)         # 注入 broadcast fn

Notifier — Strategy Pattern

新增通知管道不需要改 alert_engine，只要新增一個 BaseNotifier 子類並登記進 NOTIFIERS dict：

# backend/app/services/notifiers/__init__.py
NOTIFIERS = {
    "email":    EmailNotifier(),
    "telegram": TelegramNotifier(),
    "line":     LineNotifier(),
}

# backend/app/services/alert_engine.py
notifier = NOTIFIERS.get(rule.channel)
await notifier.send(rule.channel_target, subject, body)

Lifespan Context Manager

FastAPI 的 lifespan 取代舊版 @app.on_event，確保 startup/shutdown 對稱：

# backend/app/main.py
@asynccontextmanager
async def lifespan(app: FastAPI):
    await init_db()        # startup: 建表
    scanner = BLEScanner()
    await scanner.start()  # startup: 啟動背景 asyncio Task
    yield
    await scanner.stop()   # shutdown: 取消 Task，等待結束

6 關鍵架構決策

每個設計選擇背後的理由。

決策理由
HomeKit Bridge 是獨立 process HAP-python 有自己的 asyncio loop，嵌入 FastAPI lifespan 會 deadlock
SQLite（無 migration） 降低部署複雜度；溫濕度每分鐘一筆，SQLite 綽綽有餘
WebSocket broadcast 在 BLE loop 裡 讀完就推，延遲最低，不需要 polling
Alert cooldown 存在 DB restart 後 cooldown 狀態不遺失
sensor_id = NULL = 全域規則 一條規則保護所有 sensor，不需要重複建立 N 條
macOS BLE 用 UUID 而非 MAC bleak 在 macOS 回傳 UUID，這是 CoreBluetooth 的 OS 限制，不是 bug
module-level _scanner_instance 避免 API routes 和 main.py 之間的 circular import
Vite proxy /api & /ws dev 環境不需要 CORS，frontend 和 backend 看起來在同一個 origin
